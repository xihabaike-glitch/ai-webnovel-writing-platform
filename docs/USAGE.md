# AI 网文写作平台使用说明

本文面向第一次打开产品的作者、工作室负责人和产品验收人员。它不是开发文档，而是一份“怎么把一部网文从开书推进到可投稿样本”的操作说明。

## 1. 产品是做什么的

AI 网文写作平台是一个面向网文作者的写作生产工作台。它把传统写作资产和 AI 编辑部放在同一条流水线里：

1. 作者负责作品方向、正文采用、人工判断和最终发布。
2. 平台负责作品、章节、大纲、人物、世界观、伏笔、发布包和验收证据。
3. AI 模型负责候选稿、审稿、二改、资料整理、平台包装和任务回执。
4. 毒舌 PM 闸门负责拦住没有样本、没有复查、没有人工采用、没有失败修复的批量生产。

一句话：这不是聊天框，也不是一键生成整本书；它是一个让作者可控地使用 AI 写网文、审网文、改网文、投平台的生产系统。

## 2. 本地启动

在项目根目录执行：

```bash
npm install
npm run dev
```

默认访问：

```txt
http://localhost:3000
```

常用验证命令：

```bash
npm test
npm run build
```

如果需要初始化本地数据库，可执行：

```bash
npm run db:seed
```

## 3. 先认识主要页面

### `/`

产品首页，也是毒舌 PM 交付入口。适合从这里进入真实流水线验收。

### `/docs`

开发文档总览。这里可以看到产品定位、8 个核心平台、模型接口、真实作品流水线、最终验收矩阵。

### `/projects`

作品工作台。作者从这里创建作品、选择平台、选择篇幅、搭大树结构、管理章节、人物、世界观、伏笔、首日流程、前三章改写和发布包。

### `/projects/[projectId]`

单个作品的总控页。这里看一部作品是否具备继续生产的条件：大纲、章节、AI 写审改、模型路线、批量健康、发布包和平台复盘都会回到这里。

### `/tasks`

任务中心。跨项目的草稿、审稿、二改、导出、投稿修复、平台策略任务、失败恢复任务都会集中在这里。

### `/dispatch`

派单中心。用于把“谁做、用什么模型、输入是什么、输出是什么、人工验收是什么、下一步去哪”写成可复检回执。

### `/gate`

总闸门。它判断当前是否允许继续生产。缺样本、缺复查、失败率过高、成本异常、模型岗位缺失、发布包证据不足时，都会被拦住。

### `/failures`

失败修复中心。按模型配置、提示词上下文、样本重试、人工复盘拆分修复泳道。

### `/references`

参考库。包含 30+ GitHub 参考案例、AI 编辑部角色、平台执行卡和开发路径。

### `/settings/models`

模型设置。用于配置 Claude、DeepSeek、Kimi、GPT、OpenAI-compatible 或自定义兼容模型，并把模型绑定到不同写作岗位。

## 4. 推荐使用流程

### 第一步：创建作品

进入 `/projects`，在开书区域创建新作品。

建议先填写这些内容：

1. 作品名。
2. 目标平台。
3. 目标篇幅。
4. 题材类型。
5. 核心卖点。
6. 更新目标。

当前核心平台固定为 8 个：

1. 起点中文网。
2. 番茄小说。
3. 七猫。
4. 晋江文学城。
5. 知乎盐选。
6. WebNovel。
7. Royal Road。
8. Wattpad。

平台已经锁定，不需要继续扩平台。后续重点是让每个平台的写作抓手、投稿抓手和复盘指标跑通。

### 第二步：搭大树结构

作品不是从第一章乱写开始，而是先搭一棵树：

1. 根：核心卖点、主题、商业钩子。
2. 开头：第一章必须抓人。
3. 结尾：提前确定最终承诺。
4. 树干：主线推进。
5. 大枝：卷结构。
6. 小枝：支线。
7. 叶片：章节内容。
8. 土壤：平台规则、人物设定、世界观、读者预期。

推荐顺序：

1. 先写开头和结尾。
2. 再写主干。
3. 再做分支。
4. 最后填叶片和土壤。

### 第三步：准备人物、世界观和伏笔

进入作品详情页后，补齐基础写作资产：

1. 人物：目标、欲望、缺陷、变化弧光。
2. 世界观：规则、限制、代价、资源。
3. 伏笔：埋设章节、回收章节、当前状态。
4. 章节卡：本章目标、开头钩子、冲突、价值变化、章末悬念。

这些内容不是装饰。AI 后续写稿、审稿、二改都应从这些资产中取上下文。

### 第四步：只先跑首章样本

不要一开始就批量生成很多章。

推荐先做一个首章样本：

1. 生成首章候选稿。
2. 审稿检查钩子、爽点、人物弧光、平台节奏。
3. 做二改候选。
4. 作者人工采用或退回。
5. 把采用结论写成回执。

毒舌 PM 规则：候选稿没有人工采用，不算正文；没有首章样本复查，不允许扩大批量。

### 第五步：使用 AI 编辑部角色

模型不是随便聊天，而是按岗位使用。

推荐岗位分工：

1. Claude：长篇结构主编。适合人物弧光、主线支线、长上下文审稿。
2. DeepSeek：中文网文写手。适合首章样本、爽点补强、中文节奏改写。
3. Kimi：长上下文资料官。适合整理世界观、历史章节、设定和连续性。
4. GPT：海外投稿包装编辑。适合 WebNovel、Royal Road、Wattpad 的英文简介、标签和包装改写。

进入 `/settings/models` 配好模型岗位后，再让 `/tasks` 和 `/dispatch` 使用这些岗位执行任务。

### 第六步：通过任务中心处理生产工作

进入 `/tasks` 查看推荐任务。

常见任务包括：

1. 章节初稿。
2. 审稿。
3. 二改。
4. 导出。
5. 投稿资料修复。
6. 平台策略优化。
7. 失败恢复小样本。
8. 首日流程回执。

推荐批次不是越多越好。系统会根据样本、质量、成本、失败率和模型岗位给出是否允许继续生产。

### 第七步：用派单中心补回执

进入 `/dispatch`，检查每个派单是否有完整证据。

一个合格回执至少包含：

1. 执行角色。
2. 输入摘要。
3. 输出摘要。
4. 人工验收。
5. 下一步入口。
6. 通过、退回修复或暂停批量的结论。

没有回执的任务不能算闭环。

### 第八步：回总闸门验收

进入 `/gate` 做最终判断。

总闸门只允许三种结论：

1. 允许小步加码：样本、复查、质量、成本和失败回收都过线。
2. 继续观察：样本能跑通，但证据还薄，只能继续小样本。
3. 禁止放大：失败、低质量、成本异常、模型缺岗或复检不足，必须修复。

不要绕过总闸门直接批量生产。

### 第九步：失败先修复，再恢复观察

如果 `/gate` 或 `/tasks` 指向失败问题，进入 `/failures`。

优先处理：

1. 模型配置失败。
2. 提示词上下文失败。
3. 样本质量失败。
4. 人工复盘缺失。
5. 恢复观察不足。

失败清空后也不要马上放大，先跑恢复小样本。

### 第十步：生成发布包并复盘

回到 `/projects` 或作品详情页，检查发布包。

发布包至少应包含：

1. 平台定位。
2. 标题和简介。
3. 标签。
4. 样章。
5. 平台卖点。
6. 版本基线。
7. 真实反馈。
8. 复盘指标。

如果从 `/gate` 的“平台打法归档卡”回到 `/projects` 开新书，系统会带上目标平台、归档打法和来源提示，帮助复用已经通过最终交付的打法。

## 5. 不同作者怎么用

### 新人作者

先用 `/projects` 开一部短篇或中篇，不要急着写长篇。重点练三件事：

1. 第一章钩子是否抓人。
2. 前三章是否兑现承诺。
3. 目标平台是否匹配。

### 稳定更新作者

重点使用大纲树、人物、世界观、伏笔和连续性检查。每天更新前先看当前章节卡，再让 AI 做局部审稿或二改，不要让 AI 改掉整本书方向。

### 工作室作者

重点使用 `/tasks`、`/dispatch`、`/gate` 和 `/settings/models`。把每个模型当成岗位，把每次输出当成候选产物，把每次采用当成回执。

### 出海作者

重点使用 WebNovel、Royal Road、Wattpad 的平台包。中文网文逻辑不要直接翻译，要重新处理英文简介、标签、升级体系、情绪钩子和章节标题。

## 6. 平台写作注意事项

### 起点中文网

适合长线精品、宏大世界观、主线升级和 IP 改编潜力。不要只堆爽点，必须有长期结构。

### 番茄小说

适合强钩子、快节奏、密集爽点和快速反馈。第一章、前三章和标题卖点要直接。

### 七猫

适合稳定长篇、下沉市场、保底思路和连续更新。节奏要稳，情绪兑现要清楚。

### 晋江文学城

适合细腻情感、人物关系、古言现言、纯爱和强读者社区。人物动机和情绪推进要扎实。

### 知乎盐选

适合 1-15 万字短中篇、第一人称、悬疑、复仇、虐恋和反转。开头必须快速建立问题。

### WebNovel

适合玄幻、系统、升级、英文连载和全球读者。简介、标签和前期爽点要清晰。

### Royal Road

适合西幻、LitRPG、修真、升级体系和硬规则读者。系统规则和成长曲线要可信。

### Wattpad

适合欧美言情、狼人、豪门、青春情绪和社交传播。人设、情绪和标签很重要。

## 7. 常见错误

1. 把产品当聊天框用，只问 AI “帮我写一章”。
2. 没有平台目标就开始生成正文。
3. 没有结尾承诺就写长篇。
4. 首章样本没审稿就批量生成。
5. 候选稿没有人工采用就覆盖正文。
6. 失败后不看原因，直接换模型重跑。
7. 没有发布包和反馈记录，却说作品已经跑通。
8. 把新增平台当进度，忽略 8 个核心平台的真实闭环。

## 8. 最小可用验收清单

一部作品至少完成以下证据，才算跑通一轮真实样本：

1. 已创建作品，并选择目标平台和篇幅。
2. 已补开头钩子、结尾承诺、主干、分支和土壤。
3. 已有人物、世界观、伏笔或章节卡基础材料。
4. 已生成首章候选稿。
5. 已完成审稿和二改。
6. 已有人工采用或退回原因。
7. 已生成任务或派单回执。
8. 已回到总闸门复检。
9. 已处理失败修复或恢复观察。
10. 已生成发布包、版本基线和平台复盘记录。

## 9. 维护原则

后续继续开发时，优先级如下：

1. 先证明真实作品流水线，不先加新平台。
2. 先补证据闭环，不先做炫技页面。
3. 先让作者可控，不让 AI 自动覆盖正文。
4. 先跑小样本，不直接扩大批量。
5. 先修失败原因，不盲目换模型重试。
