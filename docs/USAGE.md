# AI 网文写作平台使用说明

本文面向第一次打开产品的作者、工作室负责人和产品验收人员。它不是开发文档，而是一份“怎么把一部网文从开书推进到可投稿样本”的操作说明。

## 1. 产品是做什么的

AI 网文写作平台是一个面向网文作者的写作生产工作台。它把传统写作资产和 AI 编辑部放在同一条流水线里：

1. 作者负责作品方向、正文采用、人工判断和最终发布。
2. 平台负责作品、章节、大纲、人物、世界观、伏笔、发布包和验收证据。
3. AI 模型负责候选稿、审稿、二改、资料整理、平台包装和任务回执。
4. 毒舌 PM 闸门负责拦住没有样本、没有复查、没有人工采用、没有失败修复的批量生产。

一句话：这不是聊天框，也不是一键生成整本书；它是一个让作者可控地使用 AI 写网文、审网文、改网文、投平台的生产系统。

## 2. 本地启动

在项目根目录执行：

```bash
cp .env.example .env
npm install
npm run db:seed
npm run dev
```

`cp .env.example .env` 会准备本地数据库连接配置；`npm run db:seed` 会初始化本地 SQLite 演示/种子数据。这样 fresh clone 后，`/projects`、`/references`、`/gate`、`/dispatch` 等页面能直接看到可浏览的作品、平台、任务和验收样例。

默认访问：

```txt
http://localhost:3000
```

常用验证命令：

```bash
npm test
npm run build
```

## 3. 首次使用：从 0 到第一条可验收样本

第一次打开产品，不要急着批量写正文。最稳的方式是先把“作品、章节、样本、回执、总闸门”这条闭环跑通。

推荐顺序如下：

1. 先在 `/projects` 创建一个作品。
2. 先只做一条首章样本，不做批量生产。
3. 先让 AI 产出候选稿，再让人工验收。
4. 先把回执补完整，再进入下一轮。
5. 先让总闸门确认可以继续，再考虑扩量。

最适合第一轮的目标，是得到一条可以被检查的“首章样本记录”，而不是一整本书的草稿堆。

### 第一步：进入 `/projects`

打开 `/projects` 后，先做这几件事：

1. 新建作品。
2. 选择目标平台。
3. 选择篇幅范围。
4. 写清题材类型。
5. 记录核心卖点。
6. 确认是否是首章样本试跑。

当前支持的八个平台固定为：

1. 起点中文网。
2. 番茄小说。
3. 七猫。
4. 晋江文学城。
5. 知乎盐选。
6. WebNovel。
7. Royal Road。
8. Wattpad。

这 8 个平台是当前产品的固定范围，不要把“再加平台”当成第一优先级。

### 第二步：先建故事树，再写正文

作品不是从第一章乱写开始，而是先搭一棵树：

1. 根：核心卖点、主题、商业钩子。
2. 开头：第一章必须抓人。
3. 结尾：提前确定最终承诺。
4. 树干：主线推进。
5. 大枝：卷结构。
6. 小枝：支线。
7. 叶片：章节内容。
8. 土壤：平台规则、人物设定、世界观、读者预期。

推荐顺序：

1. 先写开头和结尾。
2. 再写主干。
3. 再做分支。
4. 最后填叶片和土壤。

这样做的原因很简单：首章样本要验证的是“这个故事值不值得继续写”，不是“这一章有没有把所有设定说完”。

### 第三步：补齐最少的故事资产

进入作品详情页后，先补这些基础资产：

1. 人物：目标、欲望、缺陷、变化弧光。
2. 世界观：规则、限制、代价、资源。
3. 伏笔：埋设章节、回收章节、当前状态。
4. 章节卡：本章目标、开头钩子、冲突、价值变化、章末悬念。

这些内容不是装饰。后面的审稿、二改、发布包检查，都会依赖这些上下文。

### 第四步：只先跑一条首章样本

第一轮只跑一条首章样本，别先开批量。

推荐流程：

1. 让 AI 生成首章候选稿。
2. 在任务中心或派单里记录执行信息。
3. 进行审稿，看钩子、爽点、人物弧光、平台节奏。
4. 再做二改候选。
5. 作者人工采用或退回。
6. 把采用结论写入回执。

毒舌 PM 的规则是：没有人工采用的稿子，只能算候选稿；没有首章样本复查，不允许扩大到批量生产。

## 4. 页面级操作手册

本节按页面说清楚每个入口怎么用。

### `/projects`

这里是作品工作台，第一步先从这里开始。

你可以在这里：

1. 创建作品。
2. 选择目标平台和篇幅。
3. 建故事树。
4. 管人物、世界观、伏笔、章节卡。
5. 进入单个作品的总控页。

新手最容易犯的错误，是只在这里填个标题就离开。正确做法是至少把“平台、篇幅、卖点、开头、结尾”一起定下来。

### `/projects/[projectId]`

这是单个作品的总控页。

它用来查看一部作品是否具备继续生产的条件，重点关注：

1. 大纲是否完整。
2. 章节资产是否齐备。
3. 首章样本是否已经跑过。
4. 发布包和回执是否有记录。
5. 失败修复是否完成。

如果你不知道下一步该做什么，先回到这个页面看作品当前缺什么。

### `/tasks`

这里是任务中心。

适合处理：

1. 章节初稿。
2. 审稿。
3. 二改。
4. 导出。
5. 投稿资料修复。
6. 平台策略优化。
7. 失败恢复小样本。
8. 首日流程回执。

它的用法很直接：先看任务是什么，再看输出是不是候选稿、建议稿、回执或修复结果，不要把未验收内容直接当正文。

### `/dispatch`

这里是派单中心，用来把一次 AI 工作写成可以复查的回执。

派单时要明确：

1. 谁执行。
2. 用什么模型角色。
3. 输入是什么。
4. 输出是什么。
5. 人工验收标准是什么。
6. 下一步去哪里。

### `/gate`

这里是总闸门。

它决定当前是否允许继续生产，重点看：

1. 样本是否存在。
2. 是否已经复查。
3. 质量是否过线。
4. 成本是否正常。
5. 失败是否已回收。
6. 模型岗位是否齐备。
7. 发布包证据是否完整。

### `/failures`

这里是失败修复中心。

如果样本没过、上下文错了、模型选错了、回执缺了，就先在这里修复，再去跑恢复小样本。

### `/references`

这里是参考库。

适合查看：

1. 参考案例。
2. AI 编辑部角色说明。
3. 平台执行卡。
4. 开发与验收路径。

### `/settings/models`

这里是模型设置页。

用于配置 Claude、DeepSeek、Kimi、GPT、OpenAI-compatible 或自定义兼容模型，并把它们绑定到不同写作岗位。

## 5. 首章样本怎么跑

首章样本的目标，不是“一口气写完一卷”，而是验证这部书的第一轮可行性。

建议这样跑：

1. 先定故事树和章节目标。
2. 只生成一个首章候选稿。
3. 做一次审稿，重点看钩子、信息量、情绪推进、平台节奏。
4. 再做一次二改候选。
5. 作者做最终采用或退回。
6. 记录回执和失败点。

首章样本通过后，再考虑第二章、第三章和后续批量。

## 6. AI 编辑部岗位怎么配

模型不是随便聊天，而是按岗位使用。岗位配置的原则是：AI 负责候选产物，作者负责采用与否。

推荐的岗位分工如下：

1. Claude：长篇结构主编，适合人物弧光、主线支线、长上下文审稿。
2. DeepSeek：中文网文写手，适合首章样本、爽点补强、中文节奏改写。
3. Kimi：长上下文资料官，适合整理世界观、历史章节、设定和连续性。
4. GPT：海外投稿包装编辑，适合 WebNovel、Royal Road、Wattpad 的英文简介、标签和包装改写。
5. OpenAI-compatible：兼容型编辑位，适合你已有兼容接口、想保持统一接入方式时使用。

在 `/settings/models` 里，把每个模型绑定到清晰的岗位，比“让一个模型干所有活”更稳。

可参考的分配思路是：

1. 结构和审稿交给长上下文岗位。
2. 中文改写交给中文写作岗位。
3. 资料整理交给资料官岗位。
4. 英文平台包装交给海外编辑岗位。

## 7. 任务回执怎么填

一份合格的任务回执，核心是让别人能复查这次派单到底做了什么。

至少要写清这些证据：

1. 执行角色。
2. 输入摘要。
3. 输出摘要。
4. 人工验收结果。
5. 下一步入口。
6. 通过、退回修复或暂停批量的结论。

如果是首章样本，还建议补上：

1. 本次采纳了哪些修改。
2. 哪些点仍然需要作者判断。
3. 是否允许进入下一轮。
4. 是否需要回到 `/failures` 修复。

判断标准很简单：没有足够证据，回执就不完整；回执不完整，这次派单就不能算闭环。

## 8. 总闸门怎么看

总闸门不是摆设，它是决定能不能扩量的地方。

你通常会看到三类结论：

1. 允许小步加码：样本、复查、质量、成本和失败回收都过线，可以小幅加量。
2. 继续观察：样本已经能跑通，但证据还薄，只适合继续小样本。
3. 禁止放大：失败、低质量、成本异常、模型缺岗或复检不足，必须先修复。

实际判断时，可以按这个顺序看：

1. 先看样本是否真实存在。
2. 再看人工是否已验收。
3. 再看失败是否已修复。
4. 再看回执是否完整。
5. 最后再决定要不要扩量。

只要前面的证据薄，就先停在“继续观察”。

## 9. 失败修复怎么处理

失败出现时，不要直接换模型重跑，更不要跳过复查。

优先处理顺序建议是：

1. 模型配置失败。
2. 提示词上下文失败。
3. 样本质量失败。
4. 人工复盘缺失。
5. 恢复观察不足。

在 `/failures` 里把问题拆开后，再跑一个恢复小样本。恢复样本通过以后，才考虑回到正常节奏。

### 常见修复动作

1. 调整模型岗位。
2. 补齐上下文。
3. 收窄任务范围。
4. 重新做审稿标准。
5. 重做回执。

## 10. 发布包怎么检查

发布包不是“写完了”的同义词，它是“可被投递或归档”的证据集合。

检查发布包时，至少确认这些内容：

1. 平台定位。
2. 标题和简介。
3. 标签。
4. 样章。
5. 平台卖点。
6. 版本基线。
7. 真实反馈。
8. 复盘指标。

如果是海外平台，还要额外确认英文包装是否和目标平台匹配。

检查时关注两件事：

1. 这个包是不是能说明作品为什么适合这个平台。
2. 这个包是不是能让后面的人复查你是怎么得出结论的。

## 11. 社群共创和 7 天首章样本挑战

这个产品不仅是个人工具，也欢迎社群成员一起共建打法。

你可以这样参与：

1. 加入 AI 网文创作实验室。
2. 分享首章钩子拆解。
3. 贡献平台打法共建。
4. 提交失败样本复盘。
5. 参与 7 天 AI 网文首章样本挑战。

7 天挑战的目标很明确：不是比谁写得最多，而是比谁能把第一条可验收样本做出来。

建议节奏：

1. 第 1 天定题材、平台和卖点。
2. 第 2 天搭故事树。
3. 第 3 天做首章候选。
4. 第 4 天审稿和二改。
5. 第 5 天补回执。
6. 第 6 天过总闸门。
7. 第 7 天整理发布包和复盘。

## 12. 常见问题

### 我能不能一上来就批量生成很多章？

不建议。先跑一条首章样本，确认作品方向和证据闭环，再决定是否扩量。

### 为什么候选稿不能直接当正文？

因为 AI 输出默认是候选稿、建议稿或修复稿，不是作者最终采用。没有人工采用，就不算正文。

### 为什么总闸门总让我先观察？

因为样本、回执、复查和失败修复还不够。先把证据补齐，再考虑扩量。

### 为什么要把模型分岗位？

因为不同模型擅长的事情不一样。把结构、改写、资料整理、英文包装分开，通常更稳。

### 为什么不鼓励继续加新平台？

因为当前要先把 8 个核心平台的真实闭环跑通。平台越多，不代表能力越强。

### 任务里只有 AI 输出，没有回执，算完成吗？

不算。没有任务回执，就无法复查这次派单的依据和结果。

### 我能直接跳过 `/failures` 吗？

不建议。失败不处理，后面只会更难排查。

## 13. 最小验收清单

一部作品至少完成以下证据，才算跑通一轮真实样本：

1. 已创建作品，并选择目标平台和篇幅。
2. 已补开头钩子、结尾承诺、主干、分支和土壤。
3. 已有人物、世界观、伏笔或章节卡基础材料。
4. 已生成首章候选稿。
5. 已完成审稿和二改。
6. 已有人工采用或退回原因。
7. 已生成任务或派单回执。
8. 已回到总闸门复检。
9. 已处理失败修复或恢复观察。
10. 已生成发布包、版本基线和平台复盘记录。

## 14. 维护原则

后续继续开发时，优先级如下：

1. 先证明真实作品流水线，不先加新平台。
2. 先补证据闭环，不先做炫技页面。
3. 先让作者可控，不让 AI 自动覆盖正文。
4. 先跑小样本，不直接扩大批量。
5. 先修失败原因，不盲目换模型重试。
