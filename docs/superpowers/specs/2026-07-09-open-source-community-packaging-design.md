# Open Source and Creator Community Packaging Design

日期：2026-07-09

## 1. 背景

AI Webnovel Writing Platform 已经从计划文档推进到可运行的网页产品骨架，当前包含作品工作台、任务中心、派单中心、总闸门、失败修复中心、模型设置、参考库和使用说明。

项目下一步不是继续堆功能，而是进行一次完整的对外包装，让它能被两类人快速理解：

1. GitHub 开源用户和开发者：他们需要知道项目做什么、技术结构是什么、怎么运行、怎么贡献、Roadmap 在哪里。
2. 网文创作者和社群成员：他们需要知道这个工具为什么和普通 AI 写作工具不同，能怎么帮助开书、审稿、改稿、复盘和投稿。

本次包装目标是把项目从“开发过程中的产品”整理为“可以公开发布、可被 Star、可被试用、可被共建”的开源项目。

## 2. 包装结论

采用“双线包装”：

1. GitHub 线：把项目包装成一个严肃、可运行、可贡献的开源 AI 写作工作台。
2. 社群线：把项目包装成“AI 网文创作实验室”，吸引作者、编辑、提示词玩家、开发者和出海作者共建。

统一定位：

> 一个开源的 AI 网文创作基础设施：让作者像主编一样掌控作品，让 AI 像编辑部一样协作。

统一 Slogan：

> 让 AI 像编辑部一样协作，让作者像主编一样掌控网文生产。

英文副标题：

> Open-source AI writing workspace for serious webnovel creators.

## 3. 目标用户

### 3.1 GitHub 开源用户

需要看到：

1. 项目一句话定位。
2. 当前产品状态。
3. 快速启动方式。
4. 页面地图。
5. 技术栈。
6. Roadmap。
7. 贡献方式。
8. Issue 模板。

他们关注的是项目是否可信、是否可运行、是否有清晰边界、是否值得贡献。

### 3.2 网文创作者

需要看到：

1. 这个项目和普通 AI 聊天框有什么不同。
2. 是否支持自己关心的平台。
3. 能不能解决开头钩子、人物弧光、主线支线、伏笔、审稿、二改和投稿包问题。
4. AI 是否会夺走作者控制权。
5. 是否有社群可以交流真实样本和写作流程。

他们关注的是是否真能服务创作，而不是开发者自嗨。

### 3.3 AI 写作产品研究者

需要看到：

1. 30+ GitHub 参考案例如何转化为产品结构。
2. 多模型岗位如何组织。
3. 任务回执、PM 闸门和失败修复如何形成闭环。
4. 这个项目相对 Dify、LangChain、novelWriter、AppFlowy 等项目的差异。

他们关注的是方法论和可复用设计。

## 4. 项目表达系统

### 4.1 一句话简介

中文：

> 面向网文创作者的开源 AI 写作生产平台：大树结构、平台打法、AI 编辑部、任务回执、失败修复和投稿包一体化工作台。

英文：

> Open-source AI writing workspace for webnovel authors: story tree, platform strategy, AI editorial roles, dispatch receipts, failure recovery, and submission packages.

### 4.2 核心差异化

普通 AI 写作工具解决“帮我写一段”。

传统小说软件解决“帮我管理章节”。

通用 Agent 平台解决“帮我编排模型”。

本项目解决：

> 一部网文如何从平台选择、开篇钩子、人物弧光、章节生产、审稿二改、失败修复，到投稿包和复盘，形成一条可持续生产流水线。

### 4.3 反定位

需要明确说明项目不是什么：

1. 不是纯聊天机器人。
2. 不是一键生成百万字小说。
3. 不是自动投稿工具。
4. 不是低质量批量洗稿工具。
5. 不是用炫技页面替代真实写作闭环的演示项目。

## 5. GitHub 包装设计

### 5.1 README 结构

README 应改成对外发布型结构：

1. 项目名 + Slogan。
2. 项目一句话定位。
3. 为什么不是普通 AI 写作工具。
4. 核心能力。
5. 产品页面地图。
6. 当前支持平台。
7. 快速开始。
8. 使用流程。
9. 技术栈。
10. Roadmap。
11. 社群入口。
12. 贡献指南。
13. License。

### 5.2 README 首屏要求

首屏必须回答三件事：

1. 这是什么。
2. 给谁用。
3. 和其他 AI 写作工具有什么不同。

建议首屏文案：

```md
# AI Webnovel Writing Platform

让 AI 像编辑部一样协作，让作者像主编一样掌控网文生产。

AI Webnovel Writing Platform 是一个面向网文创作者的开源 AI 写作生产平台。它不是普通 AI 聊天框，也不是一键生成小说工具，而是把大树结构、平台打法、AI 编辑部、任务回执、失败修复和投稿包放进同一条可验收创作流水线。
```

### 5.3 需要新增或整理的文档

根目录：

1. `CONTRIBUTING.md`：贡献指南。
2. `ROADMAP.md`：对外路线图。

docs 目录：

1. `docs/POSITIONING.md`：项目定位与独特价值。
2. `docs/COMMUNITY.md`：社群说明。
3. `docs/LAUNCH_PLAN.md`：发布计划。
4. `docs/PRESS_KIT.md`：发布文案素材。
5. `docs/USAGE.md`：已存在，后续只补充入口。
6. `docs/GITHUB_INTRO.md`：已存在，后续整合到 README 和 PRESS_KIT。

GitHub 目录：

1. `.github/ISSUE_TEMPLATE/bug_report.md`
2. `.github/ISSUE_TEMPLATE/feature_request.md`
3. `.github/ISSUE_TEMPLATE/platform_rule.md`
4. `.github/ISSUE_TEMPLATE/writing_workflow.md`
5. `.github/pull_request_template.md`

### 5.4 Issue 模板定位

Bug Report：

用于报告页面、流程、模型配置、任务执行、发布包生成等问题。

Feature Request：

用于提出新功能，但需要说明使用场景，不接受只有一句“加一个功能”的需求。

Platform Rule：

用于贡献平台规则，例如番茄开篇节奏、起点长篇结构、知乎盐选反转、Royal Road 升级体系。

Writing Workflow：

用于提交真实写作流程、首章样本挑战、失败复盘和投稿包经验。

Pull Request Template：

要求说明改动类型、用户价值、测试方式和是否影响现有写作流水线。

## 6. 社群包装设计

### 6.1 社群名称

中文：

> AI 网文创作实验室

英文：

> AI Webnovel Creation Lab

### 6.2 社群定位

> 给网文作者、AI 写作产品开发者、剧情策划和内容创业者的开源实验社群。

### 6.3 社群欢迎语

```md
欢迎来到 AI 网文创作实验室。

这里不是“关键词生成爽文”的灌水群。

我们主要讨论四件事：

1. 网文创作方法：开头钩子、前三章、人物弧光、主线支线、伏笔回收。
2. 平台打法：番茄、起点、七猫、晋江、知乎盐选、WebNovel、Royal Road、Wattpad。
3. AI 工作流：Claude、DeepSeek、Kimi、GPT 如何分工参与写作。
4. 开源共建：一起打磨 AI 网文写作平台，贡献模板、规则、提示词、测试样本和产品建议。
```

### 6.4 社群规则

1. 可以讨论创作，但不要刷低质生成文。
2. 可以分享提示词，但更欢迎分享完整流程和复盘。
3. 可以吐槽 AI，但最好带样本和问题。
4. 可以提需求，但请尽量说清楚使用场景。
5. 尊重作者控制权，AI 是工具，不是替代作者的借口。

### 6.5 首批栏目

1. 首章钩子拆解。
2. 平台打法共建。
3. AI 编辑部提示词共创。
4. 失败样本复盘。
5. 投稿包互评。
6. 7 天 AI 网文首章样本挑战。

### 6.6 首批招募角色

1. 网文作者。
2. 编辑或责编。
3. AI 提示词玩家。
4. 前端或全栈开发者。
5. 出海作者。
6. 平台研究者。
7. 内容运营。

### 7.3 社群招募短文

目标：吸引作者和实践者入群。

主文案：

```md
我开源了一个 AI 网文写作平台。

它不是让 AI 替你写小说，而是帮你建立一个 AI 编辑部：有人管结构，有人写样本，有人审稿，有人二改，有人做投稿包装。

作者还是主编，AI 只是编辑部。

现在准备建立“AI 网文创作实验室”，欢迎网文作者、编辑、提示词玩家、开发者和出海作者一起共建。
```

### 7.4 短视频或动态文案

目标：用 30 秒解释项目。

```txt
我做了一个开源 AI 网文写作平台。

不是那种输入一句话就生成百万字的玩具。

它更像一个 AI 编辑部：
DeepSeek 负责中文网文节奏，
Claude 负责长篇结构和人物弧光，
Kimi 负责长上下文资料整理，
GPT 负责海外平台包装。

作者先搭大树结构：开头、结尾、主干、分支、叶片、土壤。
然后跑首章样本、审稿、二改、人工采用。
没有样本和复盘，就不允许批量生成。

项目已经开源，欢迎作者和开发者一起共建。
```

## 8. 发布节奏

### 第 1 阶段：开源亮相，1-3 天

目标：让大家知道这是一个开源 AI 网文写作平台。

动作：

1. 更新 README。
2. 发布 `docs/PRESS_KIT.md`。
3. 发布 `docs/COMMUNITY.md`。
4. 发布 GitHub Release。
5. 发布朋友圈、微信群、知乎想法和社群公告。

### 第 2 阶段：功能拆解，1 周

目标：让用户理解项目不是普通聊天框。

每日主题：

1. 大树写作法。
2. 8 个平台策略。
3. AI 编辑部岗位。
4. 首章样本、审稿、二改。
5. PM 闸门和任务回执。
6. 失败修复中心。
7. 投稿包和复盘。

### 第 3 阶段：共建招募，2-4 周

目标：让社群开始贡献内容。

招募贡献：

1. 平台规则。
2. 写作模板。
3. 首章样本。
4. 提示词。
5. 失败复盘。
6. Issue。
7. PR。

### 第 4 阶段：真实样本挑战

活动名称：

> 7 天 AI 网文首章样本挑战

玩法：

1. 每个参与者选择一个目标平台。
2. 用平台创建作品。
3. 补开头钩子、结尾承诺和大树结构。
4. 跑首章样本。
5. 做审稿和二改。
6. 记录人工采用结果。
7. 分享复盘。

## 9. Roadmap 包装

Roadmap 不应该只是技术清单，而要让作者和开发者都看懂。

建议分为四个阶段：

### Phase 1：开源可用

1. README、使用说明、贡献指南、路线图完善。
2. 页面地图和核心流程说明。
3. GitHub Issue 模板。
4. 社群发布包。

### Phase 2：真实样本闭环

1. 首章样本挑战。
2. 平台打法回执。
3. 真实作者反馈整理。
4. 失败样本库。

### Phase 3：平台模板共建

1. 番茄开篇模板。
2. 起点长篇模板。
3. 七猫稳定更新模板。
4. 晋江人物关系模板。
5. 知乎盐选反转模板。
6. WebNovel/Royal Road/Wattpad 出海模板。

### Phase 4：生态共建

1. 更多模型供应商适配。
2. 社群贡献提示词库。
3. 投稿包模板市场。
4. 写作样本评测集。
5. 作者工作室协作能力。

## 10. 成功标准

### 10.1 GitHub 包装验收

1. README 首屏能在 30 秒内说明项目价值。
2. 新用户能在 README 中找到启动方式。
3. 开发者能找到贡献指南。
4. 作者能找到使用说明。
5. Roadmap 能说明下一步方向。
6. Issue 模板能引导高质量反馈。

### 10.2 社群包装验收

1. 有清晰社群名称和欢迎语。
2. 有明确群规。
3. 有首批活动。
4. 有可复制招募文案。
5. 有作者、开发者、提示词玩家三类入口。

### 10.3 发布内容验收

1. 有 GitHub Release 文案。
2. 有知乎或公众号长文框架。
3. 有社群招募短文。
4. 有短视频或动态文案。
5. 有 7 天首章样本挑战说明。

## 11. 实施边界

本次包装只做开源和社群发布所需材料，不新增产品业务功能。

包括：

1. README 改版。
2. 文档新增和整理。
3. GitHub Issue 模板。
4. 社群发布文案。
5. Roadmap。
6. 贡献指南。

不包括：

1. 新增平台。
2. 改动数据库结构。
3. 增加模型调用逻辑。
4. 新做 UI 页面。
5. 自动发布到社交媒体。
